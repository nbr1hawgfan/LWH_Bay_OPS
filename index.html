# LWH Bay Ops Scan a pallet → see its current bay (from your existing hourly-synced Sheet) → hand off to the real Mobile Portal to actually complete the move. **This app never writes to the warehouse database or the Mobile Portal.** It's a read-only lookup + a launcher that pre-fills the pallet ID on the official move screen, so staff don't have to scan once and then retype the ID a second time. --- ## What's in this project - `Code.gs` — Apps Script backend. Reads the existing Bay Lookup Sheet (same Sheet ID as your Inventory Portal) and exposes a `lookupPallet` JSON endpoint. - `index.html` — the PWA itself: search box, camera scan (native `BarcodeDetector`, falls back to ZXing on iOS Safari), result card, and the "Move This Pallet" handoff button. - `manifest.json`, `sw.js`, `icons/` — standard PWA install support. --- ## Setup ### 1. Deploy the GAS backend 1. Create a **new** Apps Script project (separate from your other tools — this keeps it isolated while you test). 2. Paste in `Code.gs`. 3. Confirm `SHEET_ID` at the top points to your Bay Lookup Sheet (currently set to the same ID your Inventory Portal uses). 4. Deploy → **New Deployment** → Web app - Execute as: **Me** - Who has access: **Anyone** 5. Copy the deployment URL. ### 2. Point the frontend at your backend Already done — `GAS_URL` in `index.html` is set to: ``` https://script.google.com/macros/s/AKfycbz0jPhN6ZDuy_HrLsKpTMkpKbWq9YLCykxI2ybjPsjYFT3d6ZhlTuuvBmVLOvEREBnN9w/exec ``` If you ever redeploy that Apps Script project as a **new** deployment (not "manage deployments → edit"), you'll get a new URL and need to update this line. Redeploying the *same* deployment keeps the URL stable. ### 3. Confirm the move handoff parameter Also in `index.html`: ```javascript var MOVE_BASE_URL = 'http://mobile.logistics-warehouse.biz/BayMove.aspx'; var MOVE_PARAM_NAME = 'palletId'; // <-- swap this once confirmed ``` Once your developer confirms what query-string parameter `BayMove.aspx` reads on load (e.g., `palletId`, `pid`, `id`), update `MOVE_PARAM_NAME` here. That's the only line that needs to change for the handoff to work. If it turns out `BayMove.aspx` **doesn't** support a prefill parameter yet, the button will still open the page — staff will just need to type the ID in themselves, same as today. Nothing breaks; you just don't get the time-saving part yet. ### 4. Push to GitHub Pages Same as your other PWAs — push this folder to a repo, enable Pages on the root, and the app is installable from Chrome on Android. --- ## Notes for iteration - The bay data shown is only as fresh as your hourly Sheet sync — the app makes this explicit in the UI ("Bay data syncs hourly... not live") so nobody assumes it's real-time. - If/when the developer opens a real write path (API, stored proc, or staging table — see the separate proposal doc), the "Move This Pallet" button is the natural place to swap from "open BayMove.aspx" to "submit directly," without changing anything else about the app.
